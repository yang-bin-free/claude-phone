# CodeAfar 品牌、会话创建与多引擎设计

**日期：** 2026-07-19  
**状态：** 已确认
**产品名：** CodeAfar  
**副标题：** Run locally. Code from anywhere.

## 1. 决策摘要

本次设计同时解决三个已经确认的问题：

1. 产品从 `Claude Phone` 更名为 **CodeAfar**，不再暗示 Anthropic 官方产品，也不再把能力限制在 Claude 或手机。
2. Mac 端采用 Claude Desktop / Codex 类似的“新会话准备页”：目录、权限等会话参数放在输入框下方，发送第一条消息时才真正创建会话。
3. UI 和协议使用通用会话配置，底层由引擎适配器提供权限模式与执行行为。当前交付 Claude Code 适配器；未来新增 Codex 适配器时不重做界面。

本设计与端侧语音规范互相独立但会进入同一交付计划。语音识别只负责填写新会话或现有会话的输入框，不改变会话创建规则。

## 2. 产品身份与更名范围

### 2.1 用户可见名称

所有新构建统一显示 **CodeAfar**：

- macOS 应用：`CodeAfar.app`
- 菜单栏、窗口标题、关于信息与通知：`CodeAfar`
- iOS 与 Android 显示名：`CodeAfar`
- 安装包：`codeafar-macos-*`、`codeafar-android-*`、`CodeAfar.ipa`
- 命令行：`codeafar` 与 `codeafar-agent`
- 默认本地数据目录：`~/.codeafar`

面向用户的说明使用：

> CodeAfar 让你从手机或 Mac 控制运行在自己电脑上的编码智能体。  
> Run locally. Code from anywhere.

### 2.2 兼容迁移

更名不能丢失现有配对、项目、会话、模板和权限规则：

- CodeAfar 第一次启动时，如果 `~/.codeafar` 不存在但 `~/.claude-phone` 存在，则在旧应用已退出后把旧目录原子迁移为新目录。
- 已显式传入 `--data-dir` 的环境不自动迁移。
- 新版本识别旧配置中的 Claude 会话并默认标记 `provider: claude`。
- 安装脚本停止旧进程、迁移自启配置、安装 `/Applications/CodeAfar.app`，验证成功后才移除旧的 `/Applications/Claude Phone.app`。
- JavaScript 原生桥改名为 `window.codeAfar`，同时保留一版只读兼容别名 `window.claudePhone`，避免旧缓存页面启动失败。

GitHub 仓库名、当前工作区目录和 Go module 路径暂不在本次自动改名。它们依赖外部仓库重命名，应在产品构建验证后单独迁移；用户安装包和应用内不会继续展示旧品牌。

## 3. Mac 新会话体验

### 3.1 页面结构

点击侧边栏“新建会话”或按 `⌘N` 后，主区域进入未持久化的 **Session Draft**：

- 标题：`开始新的开发任务`
- 主输入框：`告诉 CodeAfar 要做什么…`
- 输入框下方的上下文栏依次显示：执行位置、工作目录、引擎（多引擎时）、权限模式、发送按钮。
- 当前只有 Claude 一个引擎时隐藏引擎选择，只显示 `Claude Code` 状态；增加 Codex 后才显示可切换控件。
- 模型控件采用同样的能力驱动规则：适配器只提供默认模型时隐藏；以后提供可选模型时显示。

聊天顶部不再永久显示“默认目录 / 严格”下拉框。已有会话的标题区只展示当前目录、引擎与权限摘要；权限可以点击修改，目录和引擎只读。

### 3.2 工作目录

Mac 端目录控件必须提供真实选择能力：

- 显示目录名和折叠后的实际路径，例如 `claude-phone · ~/free-object/claude-phone`。
- 点击后显示最近使用和已配置项目；“选择文件夹…”打开 macOS `NSOpenPanel`。
- 选择器只允许现有目录，不允许文件、相对路径或不存在的路径。
- 用户确认后把目录加入项目列表，后续 Mac 和手机均可复用。
- 没有可用目录时不显示“默认目录”，而是显示必须完成的“选择项目”。

手机无法直接浏览 Mac 的 Finder。手机新建会话时使用 Mac 已授权的项目列表；列表为空时明确提示“请先在 Mac 上添加工作目录”。

### 3.3 创建时机与会话名称

只有在以下条件同时满足后，发送按钮才可用：

- 引擎在线且支持新会话；
- 工作目录有效；
- 权限模式有效；
- 第一条消息非空。

用户按 Return 或点击发送时，客户端顺序执行：

1. 发送 `create_session`，包含目录、引擎、权限模式和确定性的会话名称。
2. 收到 `session_created` 后立即发送第一条文本消息。
3. 创建或首条消息发送失败时保留完整草稿，允许重试，不产生第二个会话。

会话名称由第一条消息本地生成：取清理空白后的前 32 个字符；空白或无法生成时回退到“目录名 · 时间”。不额外调用模型生成标题，避免浪费 token。

## 4. 参数修改规则

与官方桌面端保持一致：

- Session Draft 阶段：目录、引擎、权限模式和可用模型均可修改。
- 会话创建后：工作目录和引擎固定；更换任一项需要新建会话。
- 会话创建后：权限模式可以修改；可选模型是否可修改由适配器声明。
- 切换到已有会话再返回空 Draft 时保留选择；Draft 中已有未发送文字时，离开前提示是否丢弃。

权限切换只影响后续动作：

- 引擎空闲时立即应用。
- 引擎正在生成或执行工具时记录为 pending，并显示“将在本轮结束后应用”。
- 应用成功后再更新 UI 与持久化会话元数据；失败则恢复原值并显示原因。
- 进入完全访问类模式必须再次确认危险范围。

Claude Code 官方支持在流式会话中动态切换权限。Claude 适配器优先使用官方动态控制能力；如果当前 CLI 版本不提供所需控制通道，则只在安全空闲边界有序停止并 `--resume` 同一会话，以新模式恢复，不能丢失上下文或重复用户消息。

## 5. 通用引擎适配层

### 5.1 通用会话字段

协议中的会话配置使用：

- `provider`：稳定 ID，当前为 `claude`，未来可为 `codex`。
- `workingDir`：绝对路径，创建后固定。
- `permissionMode`：由 provider 解释的模式 ID。
- `model`：可选，由 provider 解释；空值表示 provider 默认。
- `name`：客户端确定性生成的显示名称。

旧客户端没有 `provider` 时按 `claude` 处理，保证协议向后兼容。

### 5.2 Provider Descriptor

每个适配器向 UI 和协议层提供描述，而不是让界面写死 Claude 的值：

- 引擎 ID、显示名、可用状态和不可用原因；
- 支持的权限选项：ID、用户标签、解释、危险等级、是否可在会话中切换；
- 可选模型与是否支持会话中切换；
- 新建、恢复、发送、停止、修改权限和健康检查能力；
- provider 原始错误到 CodeAfar 统一错误的映射。

当前 Claude 选项至少包含：

- `default`：每次询问；
- `acceptEdits`：自动接受编辑；
- `plan`：只读规划；
- `bypassPermissions`：完全访问，仅在 Claude 安装和策略允许时出现。

未来 Codex 不能简单复用 Claude 字符串。Codex 适配器会把自己的 sandbox、approval policy 和 approval reviewer 组合成用户可理解的选项，例如“询问批准”“自动审阅”“只读”和“完全访问”。同一 UI 负责渲染，但具体能力和风险说明来自 Codex descriptor。

### 5.3 服务端边界

引擎核心不再直接构造 Claude 命令，而是通过 `ProviderRegistry` 查找适配器。注册表初期只有 Claude：

- 未安装或不可用的 provider 不允许创建会话，并返回可操作错误。
- 会话持久化 provider ID；恢复时必须回到原 provider。
- 不允许把 Claude 会话切换成 Codex 会话，也不伪造跨 provider 上下文恢复。
- 将来新增 Codex 是独立功能，不属于本次实现范围。

## 6. 状态与错误处理

新会话 UI 使用明确状态：

- `draft`：尚未创建，输入可编辑。
- `creating`：已锁定参数，等待服务端确认。
- `ready`：会话已创建，可发送首条消息。
- `failed`：创建或首条发送失败，保留草稿并允许重试。

关键错误必须就地显示：

- 目录不存在、不可访问或不在已授权列表；
- Claude Code 未安装、版本不支持或认证失效；
- 权限模式被本机或组织策略禁用；
- 创建超时、连接断开、并发会话达到上限；
- 权限切换失败或需要等到当前轮次结束。

客户端为每次创建生成 `requestId`。重连或重试使用同一个 ID，服务端在短期幂等窗口内返回同一 `session_created`，避免“点一次却创建两个会话”。

## 7. 视觉与文案规则

本次沿用现有深色、暖橙强调色，不进行无关的全量视觉重做。重点调整信息层级：

- 左侧：品牌、明确的“新建会话”、会话历史、管理入口。
- 中间：当前会话或新会话 Draft。
- 输入框上下文栏：只放会影响这次会话或消息的参数。
- 顶部：当前状态摘要，不放新建参数。

权限文案不再使用缺少解释的“严格 / 审阅 / 信任”：

- `每次询问`：修改文件或执行受限操作前征求同意。
- `自动接受编辑`：可自动修改工作目录内文件，其他危险操作仍受限制。
- `只读规划`：分析和制定计划，不修改源代码。
- `完全访问`：跳过常规限制，仅用于隔离环境或完全信任的目录。

## 8. 测试计划

### 8.1 自动化

- Web 状态机：新建按钮只进入 Draft，不创建后端会话。
- Web 交互：Return 创建并发送；Shift+Return 换行；参数缺失禁用发送。
- Web 并发：双击、重连和超时重试不重复创建。
- Web 导航：有文字的 Draft 离开时确认，空 Draft 可直接离开。
- 原生目录选择：取消、有效目录、文件、相对路径和已删除目录。
- 协议兼容：缺失 provider 默认为 Claude；未知 provider 被明确拒绝。
- Provider Registry：创建、恢复、模式能力、错误映射和不可用状态。
- 权限更新：空闲立即生效、运行中延迟、失败回滚、危险模式确认。
- 数据迁移：旧目录完整迁移、目标已存在时不覆盖、失败可恢复、显式 data-dir 不迁移。
- 品牌契约：Mac/iOS/Android 用户可见名称与安装包中不再出现 `Claude Phone`。
- 仓库现有 Go、Web、Mac、Android 与 iOS 测试全部回归。

### 8.2 安装应用验收

- 从现有 `/Applications/Claude Phone.app` 升级后只保留 `/Applications/CodeAfar.app`，历史会话和项目仍可使用。
- 点击“新建会话”不创建空会话，选择 Finder 目录后路径清晰可见。
- 输入第一条消息并按 Return，只创建一个会话且收到真实 Claude 回复。
- 会话名称来自首条消息，不产生额外模型调用。
- 会话中切换权限后，下一次工具行为符合新模式；切换失败不会让 UI 假装成功。
- 顶部不再出现“默认目录 / 严格”创建控件。
- 旧会话可选择、恢复、停止，复制与 Return 发送等已交付功能不回归。

## 9. 本次不做

- 实现 Codex 进程适配器或 Codex 登录。
- 自动重命名远程 GitHub 仓库、当前工作区目录或 Go module。
- 多工作目录会话、远程 SSH 或云端运行环境。
- 为会话标题额外调用模型。
- 因改名清空、覆盖或静默丢弃旧数据。

## 10. 官方交互依据

- [Claude Code Desktop：开始会话](https://code.claude.com/docs/en/desktop)：在输入区域配置运行环境、项目目录、模型与权限，发送第一条任务开始会话。
- [Claude Code 权限模式](https://code.claude.com/docs/en/permission-modes)：权限选择器位于输入框附近，并支持会话中切换。
- [Codex 入门](https://openai.com/codex/get-started/)：先选择本地项目，再在任务输入区域确认运行上下文。
- [Codex approvals 与 sandbox](https://learn.chatgpt.com/docs/agent-approvals-security)：Codex 的权限由沙箱边界和审批策略共同构成，不能直接复用 Claude 的模式字符串。
