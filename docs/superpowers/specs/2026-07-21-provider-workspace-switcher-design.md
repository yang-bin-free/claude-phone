# CodeAfar 引擎工作区切换设计

**日期：** 2026-07-21  
**状态：** 已确认

## 目标

把 Claude 与 Codex 作为两个清晰、独立的会话工作区。用户在左上角选择引擎后，只看到该引擎的历史、当前会话和新建入口，避免在新建会话表单中寻找隐藏的引擎选择器。

## 桌面布局

- 左侧栏第一行使用 `Claude | Codex` 分段开关取代静态的 `CodeAfar` 品牌文字。
- 新建会话按钮缩为明确的 `＋` 按钮，与分段开关放在同一行。
- `＋` 按钮具有“在当前引擎中新建会话”的无障碍名称和悬停提示。
- 下方会话列表只展示当前选中引擎的历史。
- 产品名称继续出现在窗口标题、应用名称、关于信息和空状态文案中，不依赖侧栏占位。

## 切换行为

- 切换引擎不会改变现有会话的供应商，也不会把 Claude 上下文迁移到 Codex，反之亦然。
- 每个引擎分别记住最后打开的会话。
- 切换到另一引擎时，优先恢复该引擎最后打开且仍存在的会话。
- 如果没有可恢复的会话，则进入该引擎的新会话草稿。
- 当前聊天内容、会话历史、权限选项和新会话默认引擎必须始终与左上角选择一致。
- 点击 `＋` 时进入当前引擎的新会话草稿，不再要求用户在编辑器下方重复选择引擎。

## 可用性与异常状态

- 只有一个引擎可用时仍显示两个选项；不可用项禁用，并通过标题或辅助说明展示原因。
- 如果当前选中引擎在状态刷新后变得不可用，自动切换到第一个可用引擎。
- 两个引擎都不可用时保留切换器但禁用新建操作，并沿用现有启动错误提示。
- 服务端返回会话列表、停止会话或重连后，客户端重新应用当前引擎筛选，不显示其他引擎的会话。

## 移动端

- 移动端沿用同一工作区状态和过滤规则。
- 受宽度限制时可以将分段开关放在顶部栏，新建入口保持相邻，不改变语义和切换行为。
- iOS 原生客户端必须按供应商过滤会话并分别恢复最后选择；具体视觉实现遵循原生控件。

## 状态与数据

- 客户端维护 `activeProvider` 以及按供应商保存的最后会话 ID。
- 会话列表仍由服务端统一返回，客户端根据会话的 `provider` 字段派生当前列表。
- 新会话请求直接使用 `activeProvider`；供应商选择不再由隐藏的表单字段决定。
- 使用本地偏好存储持久化当前引擎和各引擎最后会话 ID，使应用重启后仍能恢复；这些值不包含凭据或聊天正文。

## 验收标准

1. Claude/Codex 开关与 `＋` 位于左侧栏同一行。
2. 切换到 Claude 时只显示 Claude 历史，切换到 Codex 时只显示 Codex 历史。
3. 两个引擎分别恢复各自最后打开的会话。
4. 某个引擎没有历史时进入该引擎的新会话草稿。
5. `＋` 始终使用当前引擎创建会话，编辑器区域不再出现重复的引擎选择器。
6. 不可用引擎无法选择，并能向用户说明原因。
7. 重连、停止会话和历史刷新后，开关、列表与聊天内容仍保持一致。
8. 键盘操作、无障碍名称、窄窗口和移动端布局均可用。

## 非目标

- 不支持在一个现有会话中途更换引擎。
- 不合并 Claude 与 Codex 的历史上下文。
- 不改变服务端会话归属或模型供应商协议。
